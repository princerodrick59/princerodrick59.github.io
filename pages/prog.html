State-Based Design

Rather than WPILib's Command-Based architecture, every subsystem uses a custom state machine pattern. External inputs only set the desired state — the subsystem handles all motor logic, sequencing, and readiness checks internally.

1
Predictability
Each subsystem owns its behavior entirely. No external command can interrupt motor logic mid-cycle — unexpected movements from scheduler conflicts are eliminated.
2
Debuggability
Current state logged to AdvantageKit every cycle. When a mechanism faults, we replay exactly which state it was in at the time — no guesswork.
3
Scalability
Adding a new behavior is adding one enum value and defining its logic in the periodic loop. No command-group restructuring or scheduler binding needed.
State Enums
Each subsystem defines its own enum — e.g. TurretState, ShooterState, IntakeState. Each value represents a complete behavioral mode with its own motor targets and sensor logic.
Periodic Execution
Every loop cycle the subsystem reads its current state and executes the corresponding logic directly — no scheduler, no command queuing, no external arbitration.
State Transitions
Driver controls and automation triggers simply write a new state value. The subsystem detects the change on the next cycle and handles ramping, interlocks, and readiness checks entirely on its own.
Turret Subsystem

Handles automatic target tracking and aiming, allowing the robot to score while moving across the field. Target selection is driven by robot position and alliance color — hub when on our side, depot or outpost (determined by Y-axis position) when across the field.

Automatic Target Selection
Robot position and alliance color determine the active target each cycle. Scoring mode aims at our hub; shuttling mode picks depot or outpost based on Y-axis position on the field.
Field-to-Robot Angle Conversion
Field-relative angle to the target calculated via atan2, then robot heading subtracted to get turret-relative position. When the robot rotates, the turret automatically counter-rotates to stay locked on target.
Turret Offset Compensation
The turret pivot is offset from robot center by several inches. All angle calculations compensate for this offset to prevent systematic one-sided misses regardless of shot angle.
Motion Magic Profiles
Smooth trapezoidal motion profiles for all turret movements. Angles wrapped to ±180° via modulus to prevent cable wrap issues at rotation limits.
Shot Readiness Gate
"At setpoint" detection is checked every cycle. The indexer is only permitted to fire when the turret confirms it is accurately on target — not just moving toward it.
Shooter Subsystem

Dual flywheels in leader-follower configuration with opposed motor alignment to spin in the same direction. Uses an InterpolatingDoubleTreeMap to map measured field distances to empirically tuned RPS values — distances in between are interpolated automatically. Two separate tables handle hub shots and neutral-zone shuttling.

Why Interpolation?

Fixed flywheel speeds overshoot close targets and undershoot far ones. By defining velocity values at specific measured distances — e.g. 47 RPS at 3.0m, 49 RPS at 3.5m — a shot from 3.25m automatically calculates to 48 RPS without any manual tuning in between. Air times are measured at the same known distances and stored in a parallel lookup table used by the Shot Calculator.

Shooter States

STOPMotors off — safe for disabled / end-of-match
IDLELow RPM — stays spinning to cut spin-up lag
BLUE_HUBDistance lookup table — hub shot velocity
SHUTTLEDistance lookup table — shuttling velocity

Shot Calculator — Targeting Core

"Where should I aim now so the projectile lands on target later?" Three compounding factors cause misses when shooting while moving — the Shot Calculator compensates for all three simultaneously using iterative convergence.

Three Sources of Error

Robot Translation
Movement during projectile flight shifts the effective aim point — the target drifts relative to where the robot aimed at release.
Processing Latency (~20ms)
Sensor-to-motor delay compensated by projecting the robot pose 20ms forward based on current velocity before any angle calculation begins.
Turret Offset Swing
Rotational swing from the offset pivot contributes its own velocity component to the turret's field-relative motion — included in all compensation math.

Iterative Convergence — 20 iterations/cycle

Step 1 — Latency Compensation
Project robot pose 20ms forward from the current velocity vector to find where the robot will actually be when the motor responds.
Step 2 — Velocity Calculation
Compute field-relative turret velocity, combining robot translation and all rotational components from robot spin around the offset pivot.
Step 3 — Convergence Loop
Look up air time from current distance → shift aim point opposite to turret movement during flight → recalculate distance to the new virtual target → repeat. Air time depends on distance, but distance changes when the aim point moves — the circular dependency requires iteration. Typically stable within 5–10 cycles.
Shooting Sequence & Feedback

Pulling the trigger does not fire immediately — it initiates a coordinated handshake that ensures all systems are ready before the indexer releases. This prevents premature shots from a misaligned turret or under-speed flywheels.

Step 1 — Trigger Held
Driver holds right trigger. Turret transitions to active tracking mode and the shooter begins spooling toward the distance-based target velocity.
Step 2 — Systems Confirm Ready
Turret checks "at setpoint." Shooter checks velocity within tolerance. Both must independently report Ready before the handshake advances.
Step 3 — Indexer Fires
Only when both confirmations are received does the indexer activate to feed the note into the flywheels. The gate resets immediately after the shot.

LED Visual Feedback

Green Ready to Shoot
Turret aimed + shooter at speed + trigger held. All conditions met — shot will fire.
White Shift Warning
Time warning displayed to alert driver of period transitions.
Blue Default State
Robot enabled and running normally. No active shot condition.

All states and virtual target positions logged to AdvantageKit every cycle for post-match shot analysis and fault replay.

Vision System

Two Limelight cameras provide independent AprilTag-based field localization. Rather than fusing both estimates, the system selects the best camera by distance each cycle to maximize pose stability. Four filtering layers reject bad data before any estimate reaches the pose estimator.

Field Boundary Filter
Rejects any estimated pose that falls outside the known carpet boundary. Eliminates phantom detections caused by field lighting and reflections.
Z-Error Filter
Rejects "floating" poses where the estimated robot height is inconsistent with being on the ground. Catches degenerate tag detections that produce valid-looking X/Y but wrong Z.
Ambiguity Filter
Rejects single-tag observations with high pose ambiguity — only confident, low-ambiguity detections are passed to the estimator.
Trust Scaling
Measurement uncertainty scales with Distance² and decreases as more tags are simultaneously visible. Close multi-tag observations receive maximum trust; distant single-tag observations contribute minimally.
Teleop Drive & Zones

Field-centric swerve drive with automatic zone-based speed assists. Assists are designed to reduce driver workload in tight areas and improve shot stability — not to fight the driver.

Drive Modes

NormalFull speed field-centric driving (default)
Trench SlowAll-axis speed reduction for precision navigation
Bump LockY-axis + rotation locked to center robot on bump
ShootingReduced speed for a stable aiming platform

Predictive Zone Detection

Hard zone boundaries feel jarring — the robot abruptly cuts speed the instant it crosses a line. Instead, the system looks 0.5 seconds ahead based on current velocity. If the robot will enter a zone within that window, the speed transition begins early and ramps smoothly before the boundary is reached. Drivers feel a gradual change rather than a snap.

Overrides & Fallback Systems

Every automation feature includes a manual override accessible via SmartDashboard (NetworkTables). Overrides require no code redeploy — they can be toggled between matches during pit stops or live mid-match if a sensor behaves erratically. Each fallback is designed to disable automation while preserving basic scoring capability.

SystemDashboard ToggleFallback Behavior
TurretTurret EnabledStops active tracking, holds current position — driver compensates with chassis heading
ShooterShooter EnabledReverts to 50 RPS fixed speed — driver aims from a known manual field location
VisionPer-Camera TogglesFaulty camera isolated individually; remaining cameras continue pose estimation normally
Drive AssistAssist EnabledDisables all Bump/Trench logic — full-speed driving with no zone restrictions

Graceful Degradation

Disabling the shooter's distance-based auto-aim doesn't stop the robot from scoring — it simply reverts to a predictable fixed speed the drive team can account for manually. The robot degrades to a known state, not an unknown one.

Defense Against Bad Data

Automation is only as good as its inputs. If field lighting corrupts vision or the turret begins tracking incorrectly, drivers can isolate individual components. A single sensor failure cannot cascade into the loss of the entire robot's scoring capability.
